Blog: Measuring math-glyph token compression

Math-glyph notation can cut Claude Code SPEC by 89%. Here's the receipt.

I called my SPEC "compressed" because the notation makes it fit in Claude's context. Without a benchmark, that's a marketing word — so I built one and ran 30 rows through the Claude tokenizer.

Two numbers came out:

- Minimal decoder: -29% tokens. Pure encoding savings vs the same content in fluent English.
- Canonical decoder: -89% tokens. Symbols plus what /sdd:explain pulls in via deferred sibling-context expansion.

The 60-point gap is the part most people miss. It is not the encoder doing more work in the canonical case. It is the decoder pulling in cited rows from elsewhere in SPEC so a reviewer can act on what they are reading.

A traditional human-readable spec would have to inline that context on every related row — multiplying file size and creating a drift problem the moment any cited invariant changes. Math-glyph keeps references as references and pays the inlining cost only at decode time, only for the row being read.

The benchmark: 30 rows x (1 source count + 2 decodes + 2 output counts) = 150 API calls per run against the Claude tokenizer. Results append to glyph-bench-results.json, tracked in git so the trend is visible across commits.

Two numbers, both honest about what they measure. If you are building any notation that claims to compress against prose, you probably want to measure both — not just the one that looks better.
